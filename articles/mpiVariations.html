Introduction

There are two dominant architectures in use today: distributed memory and shared memory. The shared memory provides cooperating processes with a shared memory address space that can be accessed by all processors. On the other hand, in the shared memory, parallel processing occurs through the use of shared data structures. Since the cost of a supercomputer is huge, and a pc or workstation can be bought for a decent price, everyone looks for an option to allow the same software to run on clusters of workstations and traditional supercomputers. The solution was the use of MPI (Message Passing Interface) but of course, there are other options.

MPICH

This version came from Argonne National Labs and Mississippi State University. And it works in a cluster environment by spawning processes on the nodes based on a machines file and parameters given to the mpirun command . The machines can present each of the node by name. The user can use the mpirun command and the machine will parse the machine file and on each node of the cluster a process will be spawned but this is done in the order they are listed in the machine file. The cool thing is that let's say the user wants more nodes than it's possible, well the mpirun can be made to spawn multiple processes per machine until the requested number of processes have been started. The command uses the rsh command or a functional equivalent pr using a special daemon process that will run on each node in the cluster Sadly, if the mpirun fails and the programmer did not handle the case carefully, the program may hang indefinitely. And this is a problem because then the programmer might kill the process but on the nodes there will be zombie processes that are still running wich may lead to performance degradation and it is possible that these zombie processes will consume the whole memory of a node preventing any jobs from executing.

LAM/MPI

The LAM (Local Area Multicomputer) MPI distribution spawns processes on the compute nodes in an interesting way. The user first creates a group of server daemons running on the compute nodes (they are spawned using the rsh command). And these daemons act as an "allocation" of the nodes.These LAM daemons are similar to MPICH distribution but theu run as the user invoking the job. The user must create and remove daemons through various LAM provided utilities. So this gives the user more freedom, it might cause some problems too but it all depends on the user.

PVM

PVM exactly like MPI uses rsh to spawn processes on the compute nodes. It generates daemons on each of the machines based on a machine list supplied by the user. A unique and interesting feature that the PVM has is that it allows a user to add or delete machines from a running parallel program on the fly. This would be an incredible feature to play with, resulting in many interesting experiments. PMV adds and deletes nodes from the working set by taking advantage of the pvmd that runs on each node in the active working set Again, it does provide a set of functions for parallelizing an existing source program and supports broadcasting (it sends to all processes in a group.)
